## What to Do

The final phase ensures everything is complete, properly formatted, and ready for submission. This is your last chance to review and polish your work.

### Key Activities

1. **Complete Documentation Review**
   - Review all sections for completeness
   - Check for consistency across sections
   - Verify all cross-references work
   - Proofread thoroughly

2. **Code Finalization**
   - Final code review and cleanup
   - Ensure all code is well-commented
   - Remove debug code and temporary files
   - Verify code runs correctly

3. **Evidence Compilation**
   - Ensure all screenshots are clear and labeled
   - Verify all diagrams are professional
   - Check all evidence is properly referenced
   - Organize supplementary materials

4. **Formatting and Presentation**
   - Apply consistent formatting throughout
   - Create table of contents
   - Number pages and sections correctly
   - Check file formats meet requirements

5. **Final Verification**
   - Test code on a clean system
   - Verify all files are included
   - Check submission requirements
   - Make backup copies

## Common Pitfalls

### âŒ Last-Minute Rush
**Pitfall:** Trying to complete everything in the final week.  
**Solution:** Use the full 12 months. The submission phase should be for review and polish, not creating content.

### âŒ Missing Evidence
**Pitfall:** Realizing at the end that you don't have screenshots or documentation.  
**Solution:** Collect evidence throughout the project. Take screenshots as you work. Keep all notes and emails.

### âŒ Inconsistent Formatting
**Pitfall:** Different fonts, styles, and layouts across sections.  
**Solution:** Create a style template at the start and apply it consistently. Do a formatting review before submission.

### âŒ Broken Cross-References
**Pitfall:** References to "See Section 4.2" that don't exist or are incorrect.  
**Solution:** Do a final check of all cross-references. Use your word processor's cross-reference features.

### âŒ Code Doesn't Run
**Pitfall:** Submitting code that has errors or won't run.  
**Solution:** Test your code on a clean system before submission. Include setup instructions.

### âŒ Missing Files
**Pitfall:** Forgetting to include essential files (images, database, libraries).  
**Solution:** Create a submission checklist. Test by extracting your submission on another system.

### âŒ Page/Word Count Issues
**Pitfall:** Exceeding or being well under recommended lengths.  
**Solution:** Monitor page counts throughout. Quality over quantity, but meet general guidelines.

### âŒ Poor Quality Screenshots
**Pitfall:** Blurry, cropped, or illegible screenshots.  
**Solution:** Take high-resolution screenshots. Ensure text is readable. Crop appropriately but show context.

### âŒ No Proofreading
**Pitfall:** Submitting work with typos and grammatical errors.  
**Solution:** Proofread multiple times. Use spelling/grammar checkers. Have someone else review if possible.

## Final Testing on Clean System

**CRITICAL:** Test your code on a clean computer to ensure it runs:

1. **Extract Submission**
   - Extract your submission files to a new folder
   - Verify all files are present

2. **Follow Setup Instructions**
   - Follow your own README instructions exactly
   - Note any missing steps or unclear instructions

3. **Install Dependencies**
   - Install any required libraries/frameworks
   - Document any issues encountered

4. **Run the Application**
   - Launch your application
   - Test all major features
   - Verify nothing is broken

5. **Check Test Data**
   - Verify sample data is included
   - Test with provided data
   - Confirm outputs are correct

## Common Submission Requirements

Verify these common requirements (check your specific exam board):

- **Total page limit:** Usually 40-60 pages for documentation
- **File format:** Usually PDF for documentation
- **Code format:** Usually .zip or source control repository
- **File naming:** Follow specified conventions
- **Declaration:** Sign any required declarations of originality
- **Teacher approval:** Some submissions require teacher sign-off

## Tips for Success

- **Start Early:** Don't leave submission prep to the last week
- **Triple Check:** Review everything multiple times
- **Test Everything:** Ensure code runs, links work, references are correct
- **Keep Backups:** Multiple backups in multiple locations
- **Read Requirements:** Follow submission requirements exactly
- **Submit Early:** Technical issues happen - don't submit at the deadline

## If Something Goes Wrong

### Code Won't Run
- Include comprehensive README with troubleshooting
- Document known issues
- Explain dependencies clearly
- Include test data that definitely works

### Missing Evidence
- Explain in documentation what happened
- Include what evidence you do have
- Be honest about gaps
- Focus on strong areas

### Exceeded Page Limit
- Prioritize quality over quantity
- Cut less important content
- Move some evidence to appendices
- Keep core sections strong

### Technical Submission Issues
- Submit early to allow time for problems
- Have backup submission method ready
- Document submission attempts
- Contact teacher/exam board immediately
